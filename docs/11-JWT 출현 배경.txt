[질문]
JWT(Json Web Token) 방식은 왜 생겨난걸까?

[확인]
JWT는 로그인을 더 편하게 하기 위해서가 아닌 분산 시스템에서 인증 정보를 쉽게 공유하기 위해 등장한 기술이다.

웹 인증 방식들을 정리해보자.

1. 단일 서버
    브라우저 -> 서버 접속 및 로그인 -> 세션 생성 -> JVM 메모리에 세션ID=A123456 저장 -> 쿠키 JSESSIONID=A123456 저장
    -> 다음 요청부터 서버는 JSESSIONID=A123456에서 A123456 추출 -> 특정 값이 있으면 인증되었다고 판단

    아무런 문제가 되지 않는다.

2. 다중 서버
    [문제점]
    브라우저 -> 서버 접속 -> 로드 밸런서 -> 서버 #1 -> 로그인 -> 세션 생성
                                    -> 서버 #2
                                    -> 서버 #N

    서버 #1에 로그인 하면 해당 서버에만 인증 정보를 가진 세션이 생성되므로 로드 밸런서에 의해 다른 서버로 접속했을 때
    세션 정보가 없어 다시 로그인 해야하는 문제가 발생한다.

    [해결책]
    이런 문제를 해결 하기 위해 대표적으로 두 가지 세션 관리 방식이 있다.

    (1) Sticky Session
        로그인한 서버만 사용하도록 하는 방식으로 로그인되었던 서버가 죽어버리면 끝이다.
        이를 위해서 Session Cluster 기능이 있나보다.
        생각해보면 효과가 있나?

    (2) Share Session
        세션 정보를 서버 내의 메모리에 저장하지 않고 별도의 캐시 서버 (Redis)에 저장하고 여러 서버가 공유하는 방식이다.
        이 방식이 현재 많이 사용되고 있으며 Spring Session 또한 이 방식을 채택하고 있다.

    또 다시 시스템이 거대화 되면서 작은 단위의 서비스들을 유기적으로 연결하여 개발하는 MSA(Micro Service Architecture)
    소프트웨어 방법론이 등장하면서 많은 서비스들에서 매번 인증 정보를 확인해야하는 비효율성이 발생한다.
    그래서 이렇게 복잡하게 할 바에 그냥 사용자 정보를 통채로 담고 있는 토큰을 만들어 인증을 하자는 아이디어에서 생겨난 것이
    JWT 방식이다.

    이렇게 JWT 방식을 사용하면 일일이 인증 정보 확인을 위해 다른 서버에 다녀올 필요가 없으니 효율적이라며 한동안
    "무조건 JWT가 최고다" 라는 분위기 였지만 지금은 다시 균형을 찾았다고 한다.

    [Session과 JWT 차이]
    | Session     | JWT               |
    | ----------- | ----------------- |
    | 서버가 상태 저장   | 상태 저장 안 함         |
    | 로그아웃 쉬움     | 로그아웃이 어려움         |
    | 토큰 탈취 영향 적음 | 탈취 시 만료 전까지 사용 가능 |
    | 확장성 낮음      | 확장성 높음            |
    | 서버 메모리 사용   | 서버 메모리 거의 안 사용    |
